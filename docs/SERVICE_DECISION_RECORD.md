# Service Decision Record

Status: Sprint 14 accepted for a minimal local/private scaffold

## Decision

Imprint may add a disabled-by-default local/private service scaffold that exposes only already
generated public-safe exports and public-safe operational status. The first cut is a read-mostly
service wrapper around the existing CLI/export pipeline, not a new profile builder, connector runner,
model runtime, SaaS surface, raw-corpus browser, or downstream automation engine.

## Why Service Mode Is Needed

CLI and file-drop mode remain the source of truth for batch profile compilation. Service mode is
useful only for trusted local tools that need a stable runtime contract instead of repeatedly
shelling out or guessing which generated file is current.

The service consumer is a local operator-owned tool, agent, or application that needs to ask:

- whether Imprint is alive
- which Imprint version and export contract are available
- whether the latest generated export validates
- what public-safe profile JSON or Markdown export is current
- which compatibility warnings must be surfaced before use

## Why CLI/File-Drop Alone Is Insufficient

File-drop mode does not provide a standard health, version, validation, warning, or readiness
contract. Downstream local tools can read files directly, but then each tool must duplicate export
discovery, validation, warning extraction, freshness checks, and privacy-boundary assumptions.

Service mode centralizes those checks while preserving the CLI pipeline as the batch producer.

## Allowed Consumer

Allowed first consumers:

- local operator scripts
- local-only agents and applications
- trusted private automations running on the same machine or private network

Forbidden first consumers:

- public web users
- tenants outside the operator's trust boundary
- hosted SaaS clients
- publishing systems expecting Imprint to generate, schedule, or approve content
- model runtimes expecting prompts or provider settings

## Allowed Deployment Mode

Default is disabled. When enabled, the first implementation must bind to localhost by default and
read from an explicit export directory. Reverse-proxy or private-network deployment is allowed only
as an operator-owned wrapper with authentication and no public exposure.

## Required First-Cut Endpoints

- `GET /health`
- `GET /version`
- `GET /status`
- `GET /profiles/latest`
- `GET /exports/latest.json`
- `GET /exports/latest.md`
- `GET /warnings/latest`
- `POST /jobs/dry-run`

`POST /jobs/dry-run` is authenticated and does not rebuild, ingest, publish, call providers, or read
raw sources. It reports whether the current configured export directory has a valid latest export.

## Explicitly Deferred Endpoints

- `POST /jobs/rebuild`
- connector execution
- scheduler mutation endpoints
- raw artifact browsing
- private connector configuration
- credential management
- prompt assembly
- downstream publishing workflow endpoints
- model/provider runtime endpoints

## Output Equivalence

The service returns public-safe artifacts generated by the same CLI/export code. It validates the
canonical JSON export before returning it. For the same config, source inputs, code version, and
export directory, service-returned canonical JSON must match the CLI-generated file byte-for-byte
after canonical JSON normalization or return a parity failure.

## Decision Rationale

The implementation is justified because it avoids duplicating the compiler pipeline and does not
invent a new profile class. It gives local consumers a bounded contract around already public-safe
exports while making unsafe service expansion explicit and testable.
