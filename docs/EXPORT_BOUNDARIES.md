# Export Boundaries

Status: Sprint 08 consumer-contract decision

## Decision

Core Imprint exports profile contracts, observed expression patterns, constraints, evidence metadata,
confidence, sample policy, and warnings.

Core Imprint does not export generation-ready prompts as its source of truth. Downstream adapters may
translate profile contracts into prompts, drafts, product-specific instructions, publication artifacts,
or runtime behavior outside core Imprint.

## What Imprint Can Export

Allowed core exports:

- canonical Imprint JSON
- canonical Imprint YAML, if added later from the canonical JSON contract
- human-readable report
- first-run "What Imprint Learned" summary derived from compiled profile data
- expression profile fragment for another system
- Mosvera expression overlay contract fragment
- Broadside input-constraint contract fragment
- agent/application safe lookup contract fragment
- human/CLI inspection contract fragment
- source summary and confidence metadata
- anti-pattern guidance
- platform/context fit notes when evidence-scoped
- warnings and prohibited-use metadata
- public-safe, private-local, and full-local export variants

These exports describe what the profile says. They do not own how a target model, application, or
publication system should act.

## What Downstream Systems Own

Downstream systems own:

- draft generation
- prompt assembly
- system prompts
- model-specific instruction tuning
- decoding and sampling settings
- publishing workflow
- brand enforcement
- persona simulation
- conversation behavior
- approval and editorial review
- Mosvera aesthetic-intent compilation and runtime behavior
- Broadside publication decisions and platform formatting

## Consumer Projection Policy

Consumer projections must be derived from canonical JSON. They may be target-specific views, but they
must preserve:

- no-raw-text policy
- opaque source IDs
- compatibility warnings
- limitation summaries
- confidence caveats
- bounded-interpretation policy state

Consumer projections must not add:

- raw artifact text
- source paths or private locators
- provider-specific prompts
- generation controls
- model parameters
- image generation instructions
- article-generation logic
- publishing schedules
- runtime adapter behavior

## Prompt Fragment Policy

The phrase "publishing prompt contract exporter" should stay retired from core architecture.

If prompt-like material exists, it must be:

- generated by a downstream adapter or integration layer
- derived from canonical profile fields
- labeled as consumer-specific downstream behavior
- excluded from canonical schema semantics
- tested as a projection outside profile source-of-truth semantics

## Export Review Gate

Every new exporter must pass this review question:

> Does this exporter decide how a model should generate text in a specific downstream workflow?

If yes, it is an adapter concern, not a core Imprint exporter.

Allowed core exporter behavior:

- emit structured profile fields
- emit constraints and anti-patterns
- emit evidence and confidence metadata
- emit target-consumer identifiers
- emit compatibility and warning metadata

Disallowed core exporter behavior:

- assemble system prompts
- tune prompts for a provider or model
- encode editorial workflow steps
- decide sampling or decoding behavior
- generate drafts
- compile aesthetic intent
- implement runtime adapters

## Sprint 08 Baseline

Sprint 08 implements small deterministic consumer contract helpers under `src/imprint/consumers/`.
They emit Mosvera, Broadside, agent/application, and human/CLI projections from canonical JSON.

The helpers preserve public-safe constraints and add consumer-specific tests. They do not call remote
providers, implement APIs, generate images, assemble prompts, or publish content.

## Rejected Alternatives

### Core Prompt Exporter

Rejected because it makes Imprint responsible for generation behavior and downstream model context.

### No Downstream Guidance

Rejected because profiles need to be useful. Imprint can export structured constraints and patterns;
consumers decide how to apply them.

### Consumer-Specific Profiles as Source of Truth

Rejected because it fragments the contract and makes migrations unsafe. Canonical JSON remains the
source of truth.
